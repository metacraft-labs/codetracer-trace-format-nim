# Codebase insights — codetracer-trace-format-nim

## M24a-1: steps.dat/steps.idx are now SPEC-canonical (Rust-reader-compatible)
- exec_stream.nim writes the SPEC layout (codetracer-trace-format-spec/
  seekable-zstd.md): steps.idx = [chunk_size:u32][offset:u64]... (NO
  total_events header/trailer); each steps.dat chunk's uncompressed payload is
  header-less (NO per-chunk u32 count). This is BYTE-COMPATIBLE with the Rust
  codetracer_trace_writer::step_stream writer / codetracer_trace_reader::
  step_stream_reader::StepStreamReader. Proof: tests/test_nim_step_stream_
  crossread.nim writes a Nim production bundle and shells the Rust test
  codetracer-trace-format/codetracer_trace_reader/tests/nim_step_stream_
  crossread.rs to read its steps.dat — decoded GLIs must match.
- multi_stream_writer.close() now sets has_step_stream (bit 9) in meta.dat
  (alongside has_call_stream). The FFI reader (new_trace_reader.nim
  ensureExecReader) selects framing by that flag: SET ⇒ SPEC reader, CLEAR ⇒
  legacy. initExecStreamReader takes `legacy: bool` (default false = SPEC).
- BACKWARD COMPAT: pre-M24a-1 Nim-v4 bundles never set has_step_stream, so they
  read via legacy framing automatically. The legacy reader path (per-chunk u32
  count + total_events trailer) is retained; tested by
  test_exec_stream_legacy_back_compat (hand-builds a legacy stream).
- In SPEC mode the reader recovers total_events like the Rust reader: decode the
  LAST chunk's record count, then (numChunks-1)*chunk_size + lastCount.
- Test harnesses that build a bundle by hand via initExecStreamWriter +
  writeMetaDat AND read it via the FFI reader MUST pass hasStepStream=true to
  writeMetaDat (otherwise the reader picks legacy and misparses the SPEC bytes).
  Done in test_new_trace_reader/test_reader_integration/test_reader_calls_events/
  test_cross_format/test_reader_ffi/bench_regression_suite.
- ct-print --json-events step output is UNCHANGED (the ct_print_full golden only
  flipped has_step_stream false→true; the step section is byte-identical). NOTE:
  test_ct_print_events_log_fallback caches its ct-print binary at
  /tmp/ctprint_build/ct-print and only rebuilds when src/codetracer_ct_print.nim
  is newer — editing a transitive dep (exec_stream.nim) leaves a STALE binary;
  rm -rf /tmp/ctprint_build before running it after such edits.

## ct-print reader paths (IMPORTANT, M23e context)
- ct-print has TWO independent readers:
  * `NewTraceReader` (src/codetracer_trace_writer/new_trace_reader.nim) — the
    v4 "multi-stream" reader. It sources EVERY event from the SPLIT per-kind
    streams (steps.dat / calls.dat / values.dat / events.dat + the four
    interning tables paths/funcs/types/varnames). It NEVER reads events.log.
    Its accessors read the split .dat files UNCONDITIONALLY — they do NOT gate
    on the meta.dat has_*_stream flag bits (those flags are advisory capability
    markers only; M23a–d merely surfaced them in --meta-json/--full).
  * `TraceReader` (src/codetracer_trace_reader.nim) — the legacy v2/v3 reader
    that decodes the combined `events.log`. Used as the fallback in
    codetracer_ct_print.nim `main()`.
- The two readers emit DIFFERENT --json-events formats and are NOT byte-compatible:
  v4 = lowercase type tags (path/function/step/call/value/io) with step_index,
  entry_step/exit_step, call_key, depth, separate varname/type declarations;
  legacy = capitalized (Path/Function/Step/Call/Value/Return), flat, values
  keyed by variable_id. The reprobuild engine (ctfs_trace.nim) parses the
  LOWERCASE v4 format.

## Writers are mutually exclusive on the Nim side
- `MultiStreamTraceWriter` (multi_stream_writer.nim) writes split streams +
  meta.dat, NO events.log. It stamps only has_call_stream=true (not the other
  has_*_stream bits) even though it writes all the streams.
- Legacy `TraceWriter` (codetracer_trace_writer.nim) writes events.log +
  meta.dat + meta.json + paths.json, NO split streams.
- The canonical RUST writer (codetracer-trace-format/codetracer_trace_writer)
  ALWAYS writes events.log AND additively writes split streams when the
  recorder opts in — so a real Rust bundle carries BOTH (the transition format).

## M23e-1 routing fix (codetracer_ct_print.nim main())
- Before opening with the v4 reader, ct-print inspects the layout via
  container.hasInternalFile: if steps.dat is ABSENT but events.log is PRESENT,
  it routes to the legacy reader (preferSplit=false). Otherwise (split present,
  with or without events.log) it uses the v4 reader. Without this, an
  events.log-only bundle was opened by the v4 reader and produced an empty
  event array (lazy split readers found nothing).
